선행 문서: `경쟁사_비교분석_도구_개발명세_20260605.md`(CLI 파이프라인 명세), `README.md`

## 0. 한 장 요약

| 항목 | 결정 |
|---|---|
| 입력 | **자사 샵 URL 1개 + 경쟁사 샵 URL 3개**(Qoo10) + 카테고리 라벨 |
| 처리 | 수집 → 정량 스코어링 → 디자인 자동채점(비전) → **서사 AI 초안** → **사람 검수·편집** → 렌더 |
| 출력 | **폴리시드 16슬라이드 PPTX** (PDF 자동화는 범위 외 — PowerPoint에서 사용자 내보내기) |
| 서사 방식 | **하이브리드**(AI 초안 → 사람 검수) |
| 배포 | **사내 전용 웹툴**(외부 공개·과금 없음). GitHub/Docker 배포는 **개발 샘플 완료 후 최종 단계**에 진행 |
| 핵심 신규 개발 | ① 동적 프로젝트/잡 모델 ② **report-spec(JSON) 브리지** ③ **폴리시드 PPTX 생성기 코드화** ④ 검수 웹 UI |

> **가장 중요한 한 가지:** 지금 "완성" 16슬라이드 덱은 코드가 아니라 claude.ai에서 수작업으로 만들어졌다. 서비스화의 핵심 난제는 **이 덱을 데이터+서사로부터 결정론적으로 재생성하는 생성기를 코드로 구현**하는 것이다. 이 생성기를 잇는 매개가 `report-spec`이다.

## 1. 제품 비전 & 가치 제안

- **대상 사용자:** 큐텐 재팬 입점 브랜드의 마케팅/영업 담당, 운영대행사(에이든랩 등).
- **잡 투 비 던(JTBD):** "경쟁사 대비 우리 샵이 어디서 밀리는지, 무엇을 먼저 고쳐야 하는지를 근거 있는 보고서로 빠르게 받고 싶다."
- **가치:** 수일 걸리던 경쟁 벤치마크·디자인 진단·개선 권고 보고서를, URL 4개 입력으로 **반자동 생성**. 데이터는 실측, 서사는 AI 초안 + 사람 검수로 품질 담보.
- **차별점:** 단순 크롤링 대시보드가 아니라 **컨설팅 결과물(임원 보고용 덱)**을 산출. 3-Layer(샵·상품·디자인) 프레임워크 + 5점 루브릭.

## 2. 현재 자산(As-Is)과 재사용 매핑

기존 CLI 파이프라인은 **수집·스코어링·마케팅 인텔리전스 로직이 검증 완료**되어 있고, 그대로 서비스 백엔드의 핵심 엔진으로 재사용한다.

| 기존 모듈 | 역할 | 서비스화 시 처리 |
|---|---|---|
| `shops_groups.py` | 정적 그룹(자사/경쟁) 설정 | ⮕ **동적 `projects` 테이블로 대체**(요청당 4 URL) |
| `collect.py` / `scraper.py` | Playwright 수집 오케스트레이터·DOM 파싱 | **재사용** — 잡 워커가 호출. 샵 식별을 shop_id→임의 URL 파싱으로 일반화 |
| `images.py` | 썸네일·배너·풀캡처·히어로·상세컷 | **재사용** — 산출 이미지를 오브젝트 스토리지로 |
| `db.py` | SQLite 스키마 | ⮕ **Postgres로 이전**(멀티 프로젝트·동시성) |
| `score.py` / `estimator.py` | 정량 지표·갭·추정매출 | **재사용** — 입력을 project_id 스코프로 |
| `marketing.py` | 콜라보 자동채굴·이벤트강도·톤(TONE 상수) | **재사용** + TONE 상수를 비전 자동분석으로 대체(2단계) |
| `vision.py` | 상세컷 Claude 비전 구조화 | **승격** — 디자인 루브릭 자동채점·톤 분석의 근거로 |
| `report.py` | 기본 13슬라이드 pptx·맵·갭 | 맵/갭 로직 재사용, **덱 생성은 신규 생성기로 교체** |
| (없음) "완성" 16슬라이드 덱 | 현재 ad-hoc 수작업 | ⮕ **신규 `deck/` 생성기로 코드화 (핵심 개발)** |

**검증된 데이터 한계(설계 전제):**
- 리뷰수 `999+` 상한 → 히어로 강한 샵의 리뷰볼륨·Top5파워 과소집계(상대비교용).
- 추정매출은 **2일째 스냅샷부터**(리뷰 차분) → **단발 수집(1회 방문)에서는 N/A**. → §6.2·§9 참조.
- 디자인 루브릭은 기존엔 수기/AI 초안(잠정) → 서비스에선 **비전 자동채점 + 검수 보정**.

## 3. 핵심 사용자 시나리오 (To-Be)

```
[1] 입력      사용자가 자사 URL 1 + 경쟁사 URL 3 + 카테고리명 입력 → "분석 시작"
                │
[2] 수집(잡)  비동기 워커가 4개 샵 순회 수집(샵헤더·상품·이미지·상세컷). 진행률 표시.
                │   ※ 샵당 5초+ 대기, 실패 샵 스킵·로그. 캐시 히트 시 재수집 생략.
[3] 분석      정량 스코어링 + 비전 디자인 자동채점 + 콜라보/이벤트 채굴
                │
[4] 서사 초안  Claude가 데이터·캡처를 보고 정성 슬라이드 초안 작성 → report-spec에 채움
                │
[5] 검수·편집  사용자가 웹에서 서사 텍스트·루브릭 점수·샵 라벨을 검토·수정
                │
[6] 렌더      확정된 report-spec → 폴리시드 16슬라이드 PPTX 생성
                │
[7] 다운로드   PPTX 다운로드. (PDF는 PowerPoint에서 내보내기 — 갤러리 슬라이드 정상 렌더 보장)
```

핵심 UX 원칙: **수집·분석은 길다(수 분)** → 잡 기반 진행률 + 이메일/알림. **검수는 빠르다** → 한 화면에서 서사·점수 인라인 편집.

### 4-2. 동기/비동기 경계
- **동기(즉시 응답):** 프로젝트 생성, 잡 상태 조회, report-spec 읽기/수정, PPTX 다운로드 링크 발급.
- **비동기(잡 워커):** 수집(Playwright), 비전 채점, 서사 AI 초안, PPTX 렌더. → 모두 큐로 분리. 단계별 상태(`pending→collecting→scoring→drafting→review→rendering→done|failed`).

### 4-3. 기술 스택(권장)

## 5. report-spec — 하이브리드를 잇는 단일 계약(핵심)

데이터(자동)와 서사(AI 초안→사람 수정)를 **하나의 JSON 문서**로 합친다. 파이프라인이 채우고, 사람이 검수에서 고치고, 생성기가 소비한다. **덱 생성기는 오직 이 spec만 보고 결정론적으로 렌더**한다.

```jsonc

  "category_label": "기초 스킨케어",
  "generated_at": "2026-06-12",
  "shops": [
    { "role": "own",        "id": "biohealboh_official", "name": "BIOHEAL BOH",
      "url": "https://www.qoo10.jp/shop/biohealboh_official" },
    { "role": "competitor", "id": "anua", "name": "Anua", "url": "..." }
    // ... 경쟁 3개
  ],

  // ── 데이터 슬라이드(자동, 편집 잠금 또는 보정만) ──
  "metrics": {
    "biohealboh_official": { "sku": 41, "price_median": 2450, "review_volume": 7909,
      "top5_power": 0.63, "set_ratio": 0.20, "jp_limited": 0, "satisfaction": 94,

    "biohealboh_official": { "thumbnail": 4, "shop_main": 4, "localization": 3,

              "rubric_bar": "...", "gap_bar": "..." },
  "marketing": { "collabs": [...], "event_intensity": [...], "tone": { "biohealboh_official": "..." } },

```

설계 효과:
- **AI/사람/렌더의 책임 분리**가 명확 → 각 단계 독립 테스트·재실행 가능.

## 6. 기능 요구사항(FR)

### 6-1. 입력 & 프로젝트 (FR-IN)
- URL 4개(자사1·경쟁3) 형식 검증: `https://www.qoo10.jp/shop/{slug}` 패턴. slug 자동 추출.
- 카테고리 라벨, 자사/경쟁 역할 지정. 경쟁사 **3개 고정(MVP)**, 추후 2~5 가변.
- 중복·동일 URL·죽은 샵 검출. 프로젝트 저장·재실행.

### 6-2. 수집 워커 (FR-COL) — 기존 `collect/scraper/images` 재사용
- 4개 샵 순회: 샵헤더 스냅샷(`shop_snapshots`) + 상품(`snapshots` 파생: is_set/is_jp_limited/discount_pct) + 이미지(메인배너·풀캡처·히어로) + 상세컷(상위 N, 선택).
- **매너 가드레일:** 샵 간 ≥5초, 동시성 제한, 실패 샵 스킵·로그, 재시도 백오프.
- **캐시:** 동일 샵을 24~72h 내 재요청 시 스냅샷 재사용(중복 크롤 방지·속도·비용↓).
- **단발 수집 한계:** 추정매출은 2-스냅샷 필요 → 단발 모드에서는 `est_revenue=null`, 보고서에서 해당 지표 자동 생략 또는 "N/A(단발수집)" 처리. (옵션: 프로젝트를 24h 간격 2회 자동 재방문해 추정매출 활성화)

### 6-3. 스코어링 (FR-SCORE) — 기존 `score.py` 재사용
- 정량 지표·그룹 내 순위·갭 산출(project 스코프). 4개 샵 기준으로 1위 리더 산정.

### 6-4. 디자인 자동채점 (FR-DESIGN) — `vision.py` 승격
- 샵 메인 캡처·상세컷을 Claude 비전으로 5개 항목(썸네일·샵메인·현지화·프로모설계·일관성) 채점 → `design_scores(scorer='vision_draft')`.
- **검수 단계에서 사람이 점수 보정**(하이브리드). 최종 `scorer='human'` 우선.

### 6-5. 서사 AI 초안 (FR-NARR)
- 입력: metrics·gap·design·marketing + 캡처 + 카테고리. 
- Claude가 `narrative.*`(요약 발견·시장컨텍스트·SWOT·권고·로드맵) 작성, **출처 문구 포함**. 한국어, 보고서 톤.
- 가드레일: 수치는 spec의 데이터만 인용(환각 방지), 외부 리서치는 "출처 표기 + 검수 필요" 플래그.

### 6-6. 검수·편집 UI (FR-REVIEW)
- report-spec을 폼으로 렌더: 서사 텍스트 인라인 편집, 루브릭 점수 슬라이더, 샵 표시명·갤러리 캡션 수정, KPI 카드 확인.
- "승인(approve)" 시 spec 동결 → 렌더 큐로.

### 6-7. 폴리시드 PPTX 생성기 (FR-DECK) — **신규 핵심**
- 입력: 승인된 report-spec. 출력: 16슬라이드 16:9 PPTX.
- **디자인 시스템 코드화:** 컬러 팔레트(퍼플 `#2B1B4D`·핑크 `#E0457B`·테이블 zebra 등), 폰트, 슬라이드 레이아웃·카드·표·KPI칩 컴포넌트를 재사용 함수로.
- **샵 수 가변 레이아웃:** 1+3=4개 기준으로 스코어카드 행·갤러리 카드·차트 포인트·이벤트표 행을 **동적 배치**(과거 "7개를 6칸에" 같은 오버플로/누락 방지 — 행/카드 수에 따라 간격·칸 자동계산).
- **16슬라이드 매핑(데이터 vs 서사):**

| # | 슬라이드 | 소스 | 분류 |
|---|---|---|---|
| 1 | 표지 | shops·category | 데이터(자동) |

## 7. 데이터 모델 (Postgres)

기존 스키마(`products·snapshots·shop_snapshots·image_assets·design_scores·product_details·product_concepts`) 유지 + **project 스코프 컬럼**과 신규 테이블 추가.

```sql
-- 신규
CREATE TABLE projects (            -- 분석 요청 단위
  id UUID PRIMARY KEY, owner_id TEXT, category_label TEXT,
  created_at TIMESTAMPTZ, status TEXT);          -- draft|running|review|done|failed
CREATE TABLE project_shops (       -- 자사+경쟁 URL 4개
  project_id UUID, shop_id TEXT, role TEXT, url TEXT, name TEXT);
CREATE TABLE jobs (                 -- 비동기 잡 진행상태
  id UUID, project_id UUID, stage TEXT, progress INT,
  log TEXT, started_at TIMESTAMPTZ, finished_at TIMESTAMPTZ);
CREATE TABLE report_specs (         -- §5 단일 계약(버전 관리)
  project_id UUID, version INT, spec JSONB, status TEXT, -- draft|approved
  updated_by TEXT, updated_at TIMESTAMPTZ);
CREATE TABLE artifacts (            -- 산출물 포인터
  project_id UUID, kind TEXT, path TEXT, created_at TIMESTAMPTZ); -- pptx|chart|banner
-- 기존 테이블엔 project_id(또는 snapshot 캐시 키) 추가
```

## 9. 결정사항 (2026-06-12 확정 반영)

**✅ 확정**
- **배포 범위 = 사내 전용 웹툴.** 외부 공개·과금·멀티테넌시 불필요 → 인증은 사내 SSO/간단 비밀번호 또는 무인증(망 내부), 공개 보안 하드닝 최소. 준법 리스크도 "사내 분석 용도"로 크게 완화(단, 수집 매너 가드레일은 유지).
- **GitHub 배포는 마지막.** 개발 샘플(M0~M4)을 먼저 완성·검증한 뒤 최종 단계에 GitHub/Docker/CI로 배포(M5).
- **출력 = PPTX만, 서사 = 하이브리드, 입력 = 자사1+경쟁3 URL.**

**◻ 남은 선택(개발 중 결정 가능)**
1. **호스팅:** 사내 단일 서버/VM(Docker Compose) 권장(내부 전용이라 충분). Playwright 때문에 **컨테이너 워커는 필수**.
2. **추정매출:** 단발 수집에서 생략(N/A) vs **24h 간격 2회 자동 방문**으로 활성화(잡 지연↑). → MVP는 N/A 권장.
3. **경쟁사 수:** 3 고정으로 시작하되 레이아웃은 2~5 가변으로 설계(어차피 필요).
4. **디자인 자동채점:** 비전 채점을 기본값, **검수 필수 게이트**로 사람이 최종 확정.

개발 샘플의 **백엔드 파이프라인(M0~M3 핵심)을 구현·검증 완료**. 웹 UI(M4)는 다음 단계.

| **M0 덱 생성기** | `deck/`(theme·components·charts·slides·build) — spec→16슬라이드 PPTX, **샵수 동적**(7샵/4샵 검증) | ✅ 완료 |

| **M2 AI 서사** | `draft_narrative.py` — `claude-opus-4-8`+구조화출력으로 narrative 채움(환각0·`.env` 키) | ✅ 라이브 검증 |
| **M4 웹 검수 UI** | `web/app.py`(FastAPI) — 그룹/URL 선택→build_spec→narrative 폼 편집·AI초안·승인→PPTX 다운로드 | ✅ 완료(curl+스크린샷 검증) |
| **M3 API/잡/워커** | FastAPI·RQ·Postgres·컨테이너 워커 (URL 라이브수집·멀티유저용) | ⬜ 확장 시 |
| **M5 GitHub 배포** | Docker·CI/CD | ⬜ (다음 — 샘플 완성됨) |

**현재 동작하는 전체 파이프라인 (사내 CLI):**
```bash
# 1) 수집(이미 돼 있으면 생략) → 2) 데이터 spec 자동조립 → 3) AI 서사 → 4) PPTX
python build_spec.py   --group makeup --out samples/spec_makeup_auto.json
python draft_narrative.py --spec samples/spec_makeup_auto.json --pptx out/drafted_makeup.pptx
# (draft 생략 시 build_spec --pptx 로 placeholder 서사 덱만 즉시 생성도 가능)
```
- 출력 PPTX는 PowerPoint에서 열어 검토/내보내기(갤러리 슬라이드 정상 렌더).
- `ANTHROPIC_API_KEY`는 루트 `.env`(gitignore됨). 서사 1회 ~$0.11.

**M4 착수 시 첫 화면 흐름(권장):** 그룹/URL 선택 → `build_spec` 실행 → narrative를 폼으로 렌더(텍스트·SWOT·권고 인라인 편집) → "승인"(review.status=approved) → `deck.build` → PPTX 다운로드. report-spec이 단일 계약이라 프런트는 이 JSON만 편집하면 됨.
